Attendee Search

 

Configuration for Search on Type Ahead Fields

The parameter Delay for Type Ahead Search Trigger (ms) now affects all type-ahead search fields (note that in Update 26, the parameter only worked with calendar-enabled templates, Exchange and G Suite). The parameter lets administrators configure the duration after the last keystroke entry in a search field before triggering a search call for attendees. The default is 400 ms.

This new parameter enables administrators to specify a longer duration before triggering a search event to prevent performance issues when returning results.

Fixes

Area

Description

Browse Events

Events Were Missing from Browse Events

If events took place in locations that were not included in the unauthenticated web process template, these events would not display for authenticated users in Browse Events, even though the Event Type, Status, and Room settings were set to Display to Everyday Users. The events did display from Browse Locations.

Fix: Modified functionality so that when an event is configured with the Event Type, Status, and Room settings set to Display to Everyday Users, the event will appear in Browse Events.

Internal Reference # 33741
(For EMS Internal Use Only)

Create a Reservation

Incorrectly Displaying Pop-up "Attendance must be greater than zero with rooms that have a default Setup Type"

When creating or editing a reservation, if users selected a room with a Default Setup Type configured, then the "Attendance must be greater than zero" pop-up was displayed on top of the Attendance & Setup Type dialog box. This was occurring because the defaults for the process template were being validated prior to allowing the users to input the number of attendees and the setup type.

Fix: Modified functionality so that the "Attendance must be greater than zero" pop-up is only displayed if users input an invalid attendee/setup type combination.

Internal Reference # 33758
(For EMS Internal Use Only)

Create a Reservation

Everyday User Could Create a Booking with an Invalid Email Address Format on the 1st Contact

For web process templates with the “Require Contact Email” parameter enabled, if users entered an invalid email address format in the 1st Contact Email Address field (e.g., esmith instead of esmith@gmail.com), the reservation would be created without an error message explaining the email address format was invalid.

Fix: Modified functionality so that an error message appears when users attempt to create a reservation with an invalid email address format in the 1st Contact Email Address field.

Internal Reference # 35060
(For EMS Internal Use Only)

Create a Reservation

Unable to Create Reservations When Cost Center Number Contains "&" Character

When a Cost Center Number containing an "&" character was added in EMS Desktop Client, the EMS Web App was unable to create reservations associated with that Cost Center Number. EMS Web App users would receive an error stating: "Cost Center is required and must be entered." 

Fix: Modified functionality so that Cost Center Numbers containing "&" are accepted in EMS Web App and reservations associated with those Cost Center Numbers can be made.

Internal Reference # 35112
(For EMS Internal Use Only)

Account Management >
Login

Pending Status for Returning EMS Web App Users

For EMS Web App users who were reactivated through HR Toolkit and configured to have active status on login, these users were given a pending status, instead of active, when they logged into the EMS Web App. Furthermore, EMS administrators had to manually change these everyday user account statuses from pending to active.

Fix: Modified functionality so that there is a parameter to set the status of returning users to either Active or Pending: 

  • If set to Active, then users are activated when they log into the EMS Web App.
  • If set to Pending, an account pending message will display when users log into the EMS Web App.

(For EMS Internal Use Only)

Edit Reservation
Room Request

 

Billing Reference and PO Numbers Did Not Update When Group Was Changed

When creating a reservation, if users selected a group with an associated billing reference or PO number (from the Reservation Details tab) and then changed their selection to a different group, the billing reference or PO number would not update with the new group selection.

Fix: Modified functionality so that when users select a different group, the billing reference or PO number updates and displays the correct number for the group.

Internal Reference # 36709
(For EMS Internal Use Only)

Unable to Create Reservations with Billing Reference Numbers Containing All Zeros

EMS Web App users were unable to create a reservation if they selected a billing reference number that contained all zeroes (from the Reservation Details tab). They received the error message: "Billing Reference is required and must be valid."

Fix: Modified functionality so that users can successfully create reservations with billing reference numbers that contain all zeroes.
